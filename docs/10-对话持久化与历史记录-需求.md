# 对话持久化与历史记录 — 产品需求

本文记录 **「多分支对话树会话的持久化存储与历史会话的列举、恢复调用」** 的需求，与当前实现（后端 JSON 落盘、REST 列表与加载、前端历史入口与上次会话记忆）对齐，供评审与后续迭代引用。

---

## 1. 需求概述

- **目标**：用户与助手的每次对话树会话（含节点、父子关系、分支标签、状态等）应 **可靠保存到服务端存储**，并在 **服务重启或浏览器刷新后仍可恢复**；用户应能 **浏览历史会话列表** 并 **一键切换到任意已保存会话** 继续查看或对话。
- **非目标**（当前版本不强制）：跨用户/跨设备的账号级云同步、端到端加密、会话级权限与分享链接、删除/归档/重命名会话的完整 CRUD 产品流程（可按后续需求扩展）。

---

## 2. 功能要求

### 2.1 持久化

| 项 | 说明 |
| --- | --- |
| **存储粒度** | 以 **会话（session）** 为单位；每个会话包含完整 **对话树**（节点字典、根节点列表、与树结构一致的 `children` 等）。 |
| **写入时机** | 创建会话后应落盘；对话过程中在 **关键节点变更** 后落盘（含新增用户/助手节点、流式输出过程中的节流保存、流结束或错误后的最终保存）；其他写入会话树的 API（如与分支创建相关的服务端路径）在变更后应同步持久化。 |
| **一致性** | 磁盘中的数据应能单独还原为与内存一致的会话结构，供 `GET` 全量加载；流式进行中允许节流间隔内存在轻微延迟，但结束或失败后须达到与线上一致的内容与状态。 |

### 2.2 历史列举与恢复

| 项 | 说明 |
| --- | --- |
| **列表** | 提供按 **最近更新**（或等价排序）排列的会话列表；每条至少包含 **会话标识**、**预览文案**（如首条用户消息摘要）、**节点数量**、**更新时间**，便于用户辨认。 |
| **加载** | 用户选择某条历史会话时，前端应用 **全量快照** 替换当前内存中的树，并 **绑定该会话 id** 建立 WebSocket/后续请求，使用户可继续在该会话上操作。 |
| **冷启动** | 服务端进程重启后，内存中无该会话时，应能根据会话 id **从磁盘加载** 到内存再响应查询，无需用户手动「导入文件」。 |

### 2.3 客户端体验

| 项 | 说明 |
| --- | --- |
| **刷新恢复** | 浏览器刷新后，应优先根据本地记录的 **上次会话 id** 向服务端请求快照并恢复界面；若该会话不存在或请求失败，可退化为 **新建会话**。 |
| **显式入口** | 提供 **历史会话** 入口以打开列表；提供 **新对话** 入口以创建新会话并清空当前树（与后端新 session 对齐）。 |

---

## 3. 配置与工程约定

| 项 | 说明 |
| --- | --- |
| **存储路径** | 默认使用应用数据目录下的子目录存放每会话一个 JSON 文件；可通过环境变量 **覆盖根路径**（如部署到自定义盘符或容器卷）。 |
| **流式节流** | 流式输出写入磁盘的 **最小间隔** 可通过环境变量配置，以平衡崩溃恢复与磁盘 I/O。 |
| **版本控制** | 持久化数据目录宜列入 **`.gitignore`**，避免将本地对话内容提交到仓库。 |

---

## 4. 验收要点

- 发送多轮对话并分叉后，**直接刷新页面**，应能恢复同一 `session` 下的树与内容。
- **重启后端** 后，用同一 `session_key` 请求会话详情，应返回与重启前一致的树数据。
- **历史列表** 中能出现已产生过写入的会话；点击某条后，界面树与当前会话 id 与该条一致。
- **新对话** 产生新的 `session_key`，与旧会话文件并存，互不影响。
