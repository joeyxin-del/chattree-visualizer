# 后台模型与 API 配置 — 产品需求

本文记录 **「后台（管理/设置）界面：用户自选模型、填写 API Key，并完成模型选择与调用链路的配置」** 的需求，供评审、排期与实现对齐。实现阶段的具体路由、组件名与存储格式由工程方案或 README 约定。

**说明**：文中 **「后台界面」与「设置 / 模型配置」同义**，指独立于主对话树的配置页；不要求具备「网站运营后台」的多租户管理能力。

---

## 0. 实现阶段划分（与当前开发对齐）

| 阶段 | 范围 | 与本需求的关系 |
| --- | --- | --- |
| **第一步（当前开发目标）** | 用户将开源代码克隆到本机并本地运行（如 `localhost`）；通过浏览器内的 **图形化界面** 填写 **自己的 API Key**、选择 **提供商 / 模型**；**无登录**、**不做公网部署**、**后端不按多租户设计**（隐含单用户、单运行实例）。 | 与本文 **§1.1 三条目标** 一致：配置页 + 凭据与模型 + **真实参与推理调用**。本阶段 **不实现** 账号体系与租户隔离，配置视为 **「当前实例」** 的全局设置即可。 |
| **第二步（后续）** | 登录界面、服务端按用户持久化、数据库化存储等。 | 将「当前实例一份配置」演进为 **按登录用户（或租户）隔离的配置与数据**；本文 §2–§4 的原则仍适用，存储与 API 需增加身份维度。 |

**结论**：第一步与需求 11 **在能力上是一致的**（都是「可图形化配置的 BYOK + 模型选择 + 调用链生效」）；差别仅在 **部署形态与是否多用户**，已由上表写清，避免与「后台」一词产生「多租户运营后台」的误解。

## 1. 需求概述

### 1.1 目标

- 提供 **独立的设置（后台）界面**，与主对话树视图分入口，便于集中管理 **大模型相关配置**。
- 用户能够：
  1. **选择自己要使用的模型**（在系统支持的提供商/模型列表范围内，或按可扩展方式登记；第一步可只支持既定列表 + 可选「自定义模型 ID」中的一种或组合）。
  2. **输入并保存自己的 API Key**（及与提供商相关的必要参数，如 Base URL、组织 ID 等，**第一步按需最小化**，能打通调用即可）。
  3. **将上述配置真正作用于对话与相关能力**：应用内发起模型推理时，应使用用户在设置中保存的 **提供商 + 模型 + 凭据**，而非仅前端展示。

### 1.2 非目标（按阶段）

**第一步（当前）明确不做：**

- 登录、注册、权限与 **多租户/按用户隔离** 的后端逻辑。
- 依赖 **公网域名与 HTTPS 生产部署** 的专项方案（本地 `http://localhost` 即可）。
- 将用户配置同步到云端或由平台代管密钥（BYOK 留在用户本机实例侧）。

**可整体延后（不限于第一步）：**

- 向第三方公开「模型市场」或任意用户上传自定义端点（除非单独立项）。
- API Key 的合规审计日志、计费对账等企业级能力。

### 1.3 术语

| 术语 | 含义 |
| --- | --- |
| **后台界面 / 设置页** | 用于配置模型与密钥的页面或模块；需有明确导航入口（如 `/settings`、侧栏「设置」等）。 |
| **提供商** | 如 OpenAI 兼容 API、Anthropic、Azure、本地 Ollama 等；具体支持列表以实现阶段为准。 |
| **模型** | 在选定提供商下用户可选的具体模型标识（如 `gpt-4o`、`claude-3-5-sonnet` 等）。 |
| **当前实例** | 第一步下：一次本地启动的后端进程 + 其持久化目录，隐含单一使用者，无需 `user_id`。 |

## 2. 功能要求

### 2.1 入口与基本结构

| 项 | 说明 |
| --- | --- |
| **入口** | 主应用内提供进入设置页的入口（侧栏、顶栏或独立路由等，实现时确定）。 |
| **分区** | 建议至少包含：**提供商与连接**（含 Base URL 等）、**模型选择**、**保存与（可选）校验**；多套运行环境（开发/生产）不强制。 |

### 2.2 选择模型

| 项 | 说明 |
| --- | --- |
| **可选范围** | 用户可在界面上 **从既定列表选择** 模型，或 **手动输入模型 ID**（当提供商允许任意模型名字符串时）；具体交互以实现为准。 |
| **当前生效模型** | 界面应 **明示当前已保存的默认模型**；保存后，新发起的对话或后端推理请求应使用该模型（会话级覆盖为可选扩展，第一步可不实现）。 |
| **变更生效** | 修改模型并保存后，**无需重启整个应用** 即可对后续请求生效（若后端有连接池缓存，实现上需刷新或短周期重读配置）。 |

### 2.3 输入与保存 API Key

| 项 | 说明 |
| --- | --- |
| **输入** | 提供 **密码型输入框**（默认掩码），支持粘贴；可选「显示/隐藏」切换。 |
| **保存** | 用户点击保存后，凭据应 **持久化**；第一步推荐 **由本机后端写入本地数据目录**（如单文件 JSON 或等价形式），**禁止** 将完整 Key 仅保存在浏览器 `localStorage` 且无后端参与（除非经评审接受风险并单独文档化）。 |
| **回显** | 已保存的 Key 在界面上 **不得完整明文回显**；可展示掩码（如 `sk-...xxxx`）或「已配置」状态。 |
| **更新与清除** | 支持 **覆盖保存** 新 Key；支持 **清除已保存凭据**（清除后相关调用应失败或提示未配置，行为需一致）。 |

### 2.4 配置与「真正使用」模型

| 项 | 说明 |
| --- | --- |
| **调用链** | 所有依赖大模型的能力（如对话补全、流式输出、若存在的摘要/记忆调用等）在运行时 **统一读取** 当前实例已保存的配置（提供商、Base URL、模型名、Key）。 |
| **校验** | 保存时建议提供 **「测试连接」或轻量探活**（可选但推荐）：失败时给出可读错误信息。 |
| **环境变量与设置页的优先级（第一步须约定清楚）** | 须在 **README 或实现说明** 中写明一种策略，避免行为含糊。建议：**以设置页已保存的配置为准**；若设置页未配置 Key/模型，可 **可选地** 回退到环境变量/`.env`（便于开发者克隆后快速试跑），但 **面向体验者的主路径** 应是打开设置页完成 BYOK。若采用「未配置则一律不调远程模型并提示」，也须在文档中写明。 |

## 3. 安全与隐私（原则）

| 项 | 说明 |
| --- | --- |
| **传输** | 公网部署时应使用 **HTTPS**；**第一步本地 `localhost` 使用 HTTP 可接受**。 |
| **存储** | 服务端（本机后端）存储 Key 时，宜避免明文出现在日志；落盘文件 **列入 `.gitignore`**，避免误提交。加密或系统密钥环为增强项，第一步可按最小可用实现。 |
| **权限** | 第二步引入多用户后，配置需与 **登录身份** 绑定；第一步为 **本机单用户**，不在此扩展。 |

---

## 4. 验收要点

**第一步验收：**

- 克隆代码、本地启动后，用户可在 **设置页选择模型并保存**，发起对话时 **实际使用的模型标识** 与所选一致（可通过允许的方式核对，如后端调试日志中的模型字段，且日志不得含完整 Key）。
- 用户在设置页 **填写 API Key 并保存** 后，在 Key 正确的前提下对话 **可正常返回**；Key 错误时 **有明确错误提示**，且界面不泄露完整 Key。
- **清除凭据** 后，行为与文档约定的「环境变量兜底 / 不兜底」策略一致，且 **不应静默成功却记错账户**。
- **刷新页面或重启本机后端** 后，已保存配置 **仍生效**（在实现承诺的落盘范围内）。
- **无需登录** 即可完成上述流程；同一仓库默认不包含「多租户」相关拦截逻辑。
